关于识境

识境（SENTRIX / stmem）是运行在边缘设备（Mac 等）上的视觉时空记忆系统。它以 ffmpeg 快照模式定期从摄像头获取当前画面，通过像素差筛选（MAD 阈值 50）判断是否有意义的场景变化，将有效帧送入 Gemma4 E2B 多模态大模型（支持 1~6 帧批量推理）生成结构化描述，再结合 nomic-embed-text 向量嵌入与可选人脸识别，将记忆持久化至 SQLite + LanceDB，供自然语言语义查询。

核心业务流程

1. 采帧与像素差筛选

系统启动后持续运行 ffmpeg 快照进程（每次独立 -vframes 1，避免 macOS AVFoundation 流式冻结问题），每约 2~3 秒获取一帧。CaptureController 将每帧送入 pixelDiff 快速滤镜：对比 128×72 缩略图的均值像素差（MAD），差值 ≥ 50 则判定为有效变化帧立即落地；无论差值如何，每 10 秒强制落地一帧，保证静止场景的时间线完整性。落地帧写入 data/frames/frame_NNN.jpg，同时刷新摄像头预览（data/capture/latest.jpg）。本地浏览器摄像头采用相同的像素差逻辑在前端运行。

2. 批量推理与记忆写入

落地帧进入 Batcher 队列，按如下策略调度：队列仅 1 帧时发起单帧请求；2~6 帧时在一次 API 调用中发送所有图片（批量推理）；超过 6 帧时按 6 帧一组串行处理。批量推理完成后，每帧串行执行：① 调用 nomic-embed-text 生成 768 维向量；② 通过 face-api.js 检测人脸并与已知人脸库匹配（128 维描述符，欧氏距离阈值 0.6）；③ 写入 SQLite memories 表与 LanceDB 向量索引；④ 如有新人脸同步写入 SQLite faces 表。串行写入设计避免了 SQLite 并发锁冲突。

3. 语义查询

用户输入自然语言问题后，系统通过正则解析中文时间表达（"最近N分钟"、"今天"、"昨天"、"上午/下午/晚上"等），从 LanceDB 向量索引中筛选时间范围内的候选记忆；将问题向量化后对候选集做余弦相似度排序取 Top-20；将召回记忆按时间升序排列、附加人脸身份信息，拼装完整提示词发送给 LLM；从回答中提取时间戳引用（误差 < 5s 则 score=100%）或按余弦相似度重新打分，仅展示 score ≥ 80% 的相关帧缩略图；若无帧达标则提示最高分供参考。

4. 记忆生命周期

系统每小时自动执行一次记忆清理（首次清理在启动 60 秒后）：24 小时内的记忆全量保留；超过 24 小时的按 5 分钟时间窗口压缩，每个窗口仅保留人脸数最多或描述最详细的代表帧，其余删除并同步从 LanceDB 移除对应向量；超过 7 天的记忆归档至 data/archive 目录并从向量索引清除，释放查询资源。

核心模块

视觉理解引擎（批量推理）

基于 Gemma4 E2B (2.3B) 多模态模型，支持 1~6 帧在单次 API 请求中并行理解，显著降低每帧推理延迟（6 帧批量约 5s，单帧约 6~10s）。支持 Ollama 原生接口（/api/generate）和 OpenAI 兼容接口（/chat/completions）两种后端，通过 config.json 中 inferenceServers 动态路由至不同推理机（本地 Mac / RTX 3090 / Jetson Orin 等）。

持久化存储（SQLite + LanceDB）

记忆以结构化数据存入 SQLite（better-sqlite3，WAL 模式，busy_timeout 10s），包含 memories 表（描述、时间、人脸 JSON、推理时间等）、faces 表（人物 ID、性别、年龄、128 维描述符、出现次数）和 archived_memories 归档表。向量索引使用 LanceDB（@lancedb/lancedb 持久化），支持余弦相似度检索与时间范围过滤，启动时自动对缺失向量的历史记忆执行后台回填。

人脸识别与追踪

集成 @vladmandic/face-api（SSD MobileNet + 68 点面部特征 + 128 维人脸描述符 + 年龄性别检测），实现跨帧人物追踪。同一人跨帧出现关联到同一 personId（欧氏距离 < 0.6），人脸库同时持久化至内存（faceLibrary）和 SQLite faces 表。查询时 LLM 提示词中会注明各 personId 对应的姓名/性别/年龄，避免重复计数。

浏览器摄像头采集

通过浏览器 getUserMedia API 获取本地摄像头画面，Canvas 截图后经像素差筛选（MAD 阈值 8），有效帧通过 FormData 上传至后端进行 AI 推理。需 HTTPS 访问。

技术栈

  • 运行时：Node.js v22 LTS (CommonJS)，内置 HTTP(S) 服务器 + 前端 SPA
  • 视觉模型：Gemma4 E2B 2.3B（Ollama /api/generate 或 OpenAI 兼容 /chat/completions，支持批量推理）
  • 嵌入模型：nomic-embed-text v1.5（768 维，8192 token，Ollama /api/embed）
  • 人脸识别：@vladmandic/face-api + @tensorflow/tfjs-node（SSD MobileNet + 128 维描述符）
  • 采帧：ffmpeg avfoundation 快照模式（每次独立进程，-vframes 1）
  • 场景变化检测：pixelDiff（128×72 缩略图 MAD，阈值 50；全量字节快速对比）
  • 记忆存储：better-sqlite3（SQLite，WAL 模式，busy_timeout 10s）
  • 向量索引：@lancedb/lancedb（LanceDB，持久化）
  • 上传解析：formidable
  • 前端：原生 HTML/CSS/JS（无构建工具）
  • 网络：HTTP 8080；HTTPS 8081（自签名证书）

  • src/dashboard.js — 入口：组装所有服务并启动 HTTP(S)
  • src/capture/ — CaptureController、ffmpegSnapshot 采帧源、pixelDiff 筛选
  • src/pipeline/ — FramePipeline（推理→嵌入→存储）、Batcher（批量调度）
  • src/inference/ — Gemma4 E2B 推理，支持单帧与批量
  • src/embeddings/ — nomic-embed-text 向量化
  • src/faces/ — face-api 检测与人脸库管理
  • src/memory/adapters/sqlite.js — SQLite 持久化（memories / faces / archived）
  • src/vector/adapters/lancedb.js — LanceDB 向量索引
  • src/query/ — 语义查询流水线（时间解析→召回→LLM→相关帧提取）
  • src/lifecycle/ — 记忆压缩与归档（每小时，启动后 60s 首次）
  • src/web/ — 前端 SPA（index.html + assets/）
  • data/ — frames/、capture/latest.jpg、memory.sqlite、vectors/
  • config.json — 运行时配置（推理服务器、模型、提示词、pixelDiff 参数等）
